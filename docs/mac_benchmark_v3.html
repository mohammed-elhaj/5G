What changed and why it matters

The MAC layer's headline improvement is features, not raw speed. V1 supported a single logical channel and had no scheduling. V3 supports N channels, implements the standards-compliant LCP scheduler (TS 38.321 §5.4.3.1), and reports buffer status (TS 38.321 §6.1.3.1) — all while maintaining byte-identical round-trips and backward compatibility through the V1 wrapper.

The one genuine latency improvement is the TB zero-fill fix. V1 called resize(2048, 0x00) on every TX call, zero-writing all 2048 bytes regardless of how much was used. V3 allocates without fill and only zeroes the padding region. For a 1400B packet, this saves writing ~600 bytes of zeros per call.

WSL2 timing noise prevents precise speedup ratios. Unlike the RLC layer (which showed clean 1.3–1.5× gains on native Linux), MAC's per-call time is dominated by WSL2 scheduler jitter at these latencies. The absolute numbers in this chart are directional only.

Zero regressions. All 23 original project tests pass. The old single-LCID interface is preserved as a wrapper — no callers needed updating.

Why header overhead increased

V1 packs 1 SDU per TB → 1 subheader (2–3 bytes). V2/V3 pack 2 channels × 2 SDUs each → 4 subheaders (2 bytes each = 8 bytes total, shown as 6B average). BSR adds a 2-byte MAC CE before the SDUs. This is the expected cost of multi-channel multiplexing — more channels = more subheaders, but payload bytes scale proportionally.

For a 1400-byte packet in a 2048-byte TB: 68.4% efficiency (1400B payload / 2048B TB). The remaining 31.6% is padding. This is a TB sizing issue — if TB size were tuned to actual traffic, efficiency would improve. This is a Pair D concern (variable TB sizing).
